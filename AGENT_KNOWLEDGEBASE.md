## 5. CLEAN ARCHITECTURE

Every module follows a four-layer clean architecture:

```
app/Modules/<Module>/
├── Domain/                  # Entities, Value Objects, Domain Events, Interfaces
│   ├── Models/
│   ├── Events/
│   └── Contracts/           # Repository and Service interfaces
├── Application/             # Use Cases, DTOs, Application Services
│   ├── Services/
│   ├── DTOs/
│   └── Handlers/
├── Infrastructure/          # Eloquent Models, Repositories, External APIs
│   ├── Repositories/
│   ├── Persistence/
│   └── Integrations/
├── Presentation/            # Controllers, Requests, Resources, API Routes
│   ├── Controllers/
│   ├── Requests/
│   ├── Resources/
│   └── Routes/
└── database/
    └── migrations/
```

Rules:
- Domain layer has zero external dependencies.
- Application layer depends only on Domain.
- Infrastructure implements Domain interfaces.
- Presentation depends only on Application layer (via service interfaces).
- Cross-module communication via Events only — never direct class imports between modules.

| Table | Purpose |
|-------|---------|

| `roles` | Role definitions per tenant |
| `permissions` | Permission atoms |
| `role_permissions` | Pivot |
| `user_roles` | Pivot (user → role, scoped to tenant) |

| `org_unit_user` | Users assigned to org units |
| `attachments` | Polymorphic file attachments (all entities) |

```
organization_units:
- id (PK)
- tenant_id (FK → tenants)
- parent_id (FK → organization_units, nullable)  -- recursive hierarchy
- name
- type (enum: company, division, department, branch, team, ...)
- code
- is_active
- metadata (JSON)
- timestamps
```

## 8. DATABASE DESIGN STANDARDS

### Normalization
- All tables must be in minimum 3NF / BCNF
- No repeating groups, no partial dependencies, no transitive dependencies
- Use junction/pivot tables for M:N relationships
- Polymorphic references (`entity_type` + `entity_id`) for flexible associations — used sparingly and intentionally

### Naming Conventions
- Tables: `snake_case`, plural (e.g., `order_lines`, `stock_movements`)
- Primary keys: `id` (BIGINT UNSIGNED AUTO_INCREMENT)
- Foreign keys: `<table_singular>_id` (e.g., `product_id`, `tenant_id`)
- Enums: always define allowed values explicitly in migration
- Timestamps: all tables include `created_at`, `updated_at`; soft-delete tables add `deleted_at`
- Tenant scope: every tenant-scoped table has `tenant_id` as first FK after `id`

### Constraints
- All foreign keys must have explicit DB-level constraints
- Unique keys where appropriate (e.g., `tenant_id + code`, `tenant_id + reference_number`)
- Decimal precision: monetary fields use `DECIMAL(15, 4)` minimum
- Quantity fields: `DECIMAL(15, 4)` to support fractional UoM

### Migrations
- Located in: `app/Modules/<Module>/database/migrations/`
- One migration per logical schema change
- Always rollback-safe (`down()` method fully reverses `up()`)
- Foreign key creation in correct dependency order

## 9. FINANCIAL ACCOUNTING RULES

### Double-Entry Accounting
- Every financial event produces a journal entry with balanced debit and credit lines
- `SUM(debit lines) == SUM(credit lines)` enforced at application + DB level
- No posted journal entry may be modified — only reversed via counter-entry

### Account Normal Balances

| Account Type | Increases With | Decreases With |
|-------------|---------------|----------------|
| Asset | Debit | Credit |
| Liability | Credit | Debit |
| Equity | Credit | Debit |
| Revenue | Credit | Debit |
| Expense | Debit | Credit |

### ERP-to-Accounting Event Mapping

| Business Event | Debit | Credit |
|---------------|-------|--------|
| Purchase (receive goods) | Inventory / Expense | Accounts Payable |
| Purchase payment | Accounts Payable | Bank / Cash |
| Sale (invoice) | Accounts Receivable | Revenue |
| Cost of Goods Sold | COGS | Inventory |
| Sales payment received | Bank / Cash | Accounts Receivable |
| Purchase return | Accounts Payable | Inventory / Expense |
| Sales return | Revenue (reversal) | Accounts Receivable |
| Bank fee | Bank Fee Expense | Bank Account |

## 10. INVENTORY FLOW LOGIC

### Inbound Flow (Purchasing / Receiving)
1. Purchase Order created → status: `draft`
2. PO confirmed → status: `confirmed`
3. Goods Receipt (GRN) created, linked to PO line(s)
   - Batch/Lot/Serial numbers assigned at this point
   - Location assigned
4. `stock_movements` record: type = `receipt`, linked to GRN
5. `stock_levels` updated: quantity_on_hand increases
6. Journal entry posted: Debit Inventory, Credit Accounts Payable

SMB Option: GRN without PO — `order_id` is nullable on `goods_receipts`

### Outbound Flow (Sales / Shipping)
1. Sales Order created → status: `draft`
2. SO confirmed → stock reservation created
3. Picking: items allocated from specific batch/serial/location
4. Packing → shipment record created
5. Ship → `stock_movements` record: type = `issue`
6. `stock_levels` updated: quantity_on_hand decreases
7. Journal entry posted: Debit COGS + Debit AR, Credit Inventory + Credit Revenue

SMB Option: Direct sell without SO — supported via direct shipment creation

### Allocation Strategies
- FIFO (First In, First Out) — default
- LIFO (Last In, First Out)
- FEFO (First Expired, First Out) — required for pharma/food
- Manual — user explicitly selects batch/location

### Stock Adjustment
- Cycle count variances create `stock_adjustment` records
- Approved adjustments post `stock_movements` (adjustment_in / adjustment_out)
- Journal entry: Debit/Credit Inventory Adjustment account

## 13. PERIOD-BASED ACCRUAL ACCOUNTING

Requirements:
- System supports accrual accounting — revenue and expenses recognized when earned/incurred, not when cash moves
- All journal entries are assigned to a fiscal period via `fiscal_period_id`
- Periods can be: open, closed, or locked
- Closed periods prevent new postings (enforced at application level)
- Locked periods are immutable even to administrators
- Back-dating is configurable (allow/disallow posting to prior periods)
- Year-end close process: transfer net income to retained earnings, open next fiscal year

Accrual Entry Examples:
- Revenue accrual: Debit Accrued Revenue (AR) / Credit Revenue
- Expense accrual: Debit Expense / Credit Accrued Liabilities
- Prepaid expense: Debit Prepaid (Asset) / Credit Cash; amortize each period

## 19. IMPLEMENTATION RULES FOR THE AGENT

These rules apply in every code generation and architecture decision:

1. Always read before writing. Review all existing code, migrations, and context before generating anything new.

2. Never break existing functionality. All refactors must preserve backward compatibility or provide a documented migration path.

3. One module, one concern. Never put business logic from Module A into Module B's classes. Use events for cross-module effects.

4. Interfaces first. Define repository and service interfaces in Domain layer before implementing in Infrastructure.

5. Never bypass tenant scope. Every query must filter by `tenant_id`. Use a global Eloquent scope (`TenantScope`) applied to all tenant-scoped models.

6. No raw SQL for business logic. Use Eloquent with relationships; raw SQL only for performance-critical reporting queries.

7. Every financial event = journal entry. No stock or order operation may affect financial state without a corresponding posted journal entry.

8. Migrations are forward-only in production. The `down()` method must exist but production rollbacks require a new migration.

9. All monetary values in DECIMAL(15,4). Never use float for money.

10. Soft deletes on all master data. Products, parties, accounts, warehouses — use `deleted_at` not hard deletes.

11. Audit everything. All create, update, delete, approve, reject actions must write to `audit_logs`.

12. Events over direct calls. When an order is confirmed, fire `OrderConfirmed` event — let Inventory, Finance, and Notification listeners react independently.

13. Validate at the boundary. All validation in Form Request classes (Presentation layer). Domain objects assume valid data.

14. Return DTOs, not Eloquent models, from Application Services. Keeps Presentation layer decoupled from ORM.

15. Fix all, document all. If a bug, inconsistency, or design flaw is discovered during implementation, fix it immediately and note it in a comment or changelog — do not defer technical debt.

## 2. CORE ENGINEERING PRINCIPLES

| Principle | Description |
|---|---|
| SOLID | Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion |
| DRY | Don't Repeat Yourself — eliminate all duplication |
| KISS | Keep It Simple — reduce unnecessary complexity |
| High Cohesion | Each module/class has a single, well-defined responsibility |
| Loose Coupling | Modules communicate via interfaces/contracts, never concrete dependencies |
| Separation of Concerns | Domain, Application, Infrastructure, Presentation are strictly separated |
| Interface-Driven Design | All cross-module dependencies use contracts defined in `Shared` |
| Event-Driven | Side effects are handled via domain events, not direct calls |
| ACID Compliance | All financial and inventory transactions are fully atomic |

## 4. CLEAN ARCHITECTURE

Every module follows a strict four-layer clean architecture:

```
app/Modules/<Module>/
├── Domain/               # Entities, Value Objects, Domain Events, Interfaces, Business Rules
│   ├── Models/
│   ├── Events/
│   ├── Exceptions/
│   └── Contracts/        # Repository and Service interfaces
├── Application/          # Use Cases, Commands, Queries, DTOs, Handlers
│   ├── Commands/
│   ├── Queries/
│   ├── DTOs/
│   └── Services/
├── Infrastructure/       # Eloquent implementations, external adapters, repositories
│   ├── Repositories/
│   ├── Adapters/
│   └── Listeners/
├── Presentation/         # Controllers, Requests, Resources, Routes
│   ├── Controllers/
│   ├── Requests/
│   ├── Resources/
│   └── routes/
└── database/
    ├── migrations/
    ├── seeders/
    └── factories/
```

### Layer Rules
- Domain has zero external dependencies (no Laravel, no Eloquent)
- Application depends only on Domain contracts
- Infrastructure implements Domain contracts using Eloquent / 3rd-party adapters
- Presentation depends only on Application DTOs and Services

- Each module is independently deployable, testable, and maintainable

## 6. MULTI-TENANCY

### Strategy
- Single database, tenant-scoped rows using `tenant_id` on all tenant-owned tables
- Global scope applied automatically via Eloquent model trait
- Tenant context resolved from authenticated JWT token (Passport)

### Tenant Isolation Rules
- All queries automatically filtered by `tenant_id`
- No tenant can read or write data belonging to another tenant
- Shared/global data (e.g., currency codes, country lists) lives in tenant-agnostic tables

### Hierarchical Structures
The platform natively supports recursive/nested data using adjacency list + optional materialized path:
- Product category trees
- Warehouse location hierarchies (Zone → Aisle → Rack → Shelf → Bin)
- Organization Unit trees (Company → Division → Department → Team)
- Account hierarchies in Chart of Accounts

---

## 7. DATABASE STANDARDS

### Normalization
- All tables: minimum 3NF / BCNF
- No partial dependencies, no transitive dependencies, no repeating groups

### Conventions
| Convention | Rule |
|---|---|
| Primary Keys | `BIGINT UNSIGNED AUTO_INCREMENT` (or `ULID` for distributed) |
| Foreign Keys | Explicit FK constraints with `ON DELETE` strategy defined per relationship |
| Timestamps | All tables have `created_at`, `updated_at`; soft-delete tables add `deleted_at` |
| Tenant Scope | All tenant-owned tables include `tenant_id BIGINT UNSIGNED NOT NULL` |
| Monetary Values | `DECIMAL(20,6)` — never `FLOAT` |
| Quantities | `DECIMAL(20,6)` — supports fractional units |
| Status/Type Columns | `VARCHAR` with DB-level CHECK constraints or ENUM |
| JSON Columns | Used only for truly dynamic metadata; never for queryable/indexed fields |
| Indexes | Defined explicitly; composite indexes on `(tenant_id, <key_column>)` |

### Migration Location
```
app/Modules/<Module>/database/migrations/
```
All migrations are module-scoped. The `Core` module bootstraps global/shared tables.

### 8.8 Pricing

Responsibility: Purchase and sales price lists, tiered pricing, customer/supplier-specific prices, time-bound validity.

#### Price Types
| Type | Description |
|---|---|
| `standard_purchase` | Default purchase cost |
| `standard_sales` | Default sales price |
| `tier` | Quantity-break pricing |
| `customer_specific` | Negotiated price for a specific customer |
| `supplier_specific` | Agreed cost from a specific supplier |
| `promotional` | Time-limited discount pricing |

#### Key Tables
```
price_lists           id, tenant_id, name, type (purchase|sales), currency_id,
                      is_default, valid_from, valid_to, is_active

                      valid_from, valid_to

customer_price_lists  id, customer_id, price_list_id
supplier_price_lists  id, supplier_id, price_list_id
```

Price Resolution Order (highest priority first):
1. Customer/Supplier-specific price list item (quantity match)
2. Promotional price list item (within validity window)
3. Tiered price list item (quantity-break match)
4. Default price list item
5. Product standard price (fallback)

Key Features:
- Time-bound validity on all prices
- Multi-currency price lists
- Quantity-break tiers
- Per-customer and per-supplier negotiated pricing
- Price history retained for audit

### 9.2 Returns Management

Objective: Comprehensive, auditable returns for all scenarios across both purchase and sales flows.

#### Purchase Returns (to Supplier)

Scenarios:
- Full return of a GRN
- Partial return (some lines or partial quantities)
- Return with original batch/lot/serial reference
- Return without original batch/lot/serial reference (e.g., mixed stock)
- Return of damaged/expired goods
- Return after supplier invoice already paid

Workflow:
1. Create `purchase_return` header (reference original GRN/invoice optionally)
2. Add return lines: select product/variant, quantity, condition, batch/lot/serial (optional)
3. Approve return — triggers quality check workflow (optional)
4. Disposition decision per line: `restock` | `scrap` | `return_to_vendor`
5. Post return — triggers:
   - `stock_movement` of type `return_out` (removes from warehouse)
   - Cost layer adjustment aligned with original valuation method
   - Debit note generation (AP credit)
   - Journal entry: Dr Accounts Payable / Cr Inventory or Expense

#### Sales Returns (from Customer)

Scenarios:
- Full return of a sales order / shipment
- Partial return (some lines or partial quantities)
- Return with original batch/lot/serial reference
- Return without original batch/lot/serial reference
- Return of damaged goods (not restockable)
- Return after invoice paid (refund vs credit memo)

Workflow:
1. Create `sales_return` header (reference original SO/invoice optionally)
2. Add return lines: select product/variant, quantity, condition, batch/lot/serial (optional)
3. Approve return — triggers quality check workflow (optional)
4. Disposition decision per line: `restock` | `scrap` | `quarantine`
5. Restocking fee calculated (optional, configurable per product/customer)
6. Post return — triggers:
   - `stock_movement` of type `return_in` (adds back to warehouse if restocked)
   - Cost layer re-insertion (at original cost or current cost, configurable)
   - Credit memo generation (AR debit)
   - Journal entry: Dr Sales Returns & Allowances (+restocking fee) / Cr AR

#### Return Inventory Adjustments (Valuation Alignment)
- FIFO/FEFO: Returned layers re-inserted at original layer cost
- Weighted Average: Weighted average recalculated on return
- Specific Identification: Original cost from serial/batch record used
- All layer adjustments produce `inventory_cost_layer` records with `return` reference

## 10. FINANCIAL ACCOUNTING STANDARDS

### Double-Entry Rules (The Golden Rule)
Every transaction produces balanced journal entries where:
```
Σ Debits = Σ Credits (always, without exception)
```

| Account Type | Increases With | Decreases With |
|---|---|---|
| Asset | Debit | Credit |
| Liability | Credit | Debit |
| Equity | Credit | Debit |
| Revenue | Credit | Debit |
| Expense | Debit | Credit |

### ERP Module ↔ Accounting Integration
| ERP Event | Modules Involved | Accounts Touched |
|---|---|---|
| Stock received (GRN) | Purchase → Finance | Inventory Asset ↑ / AP ↑ |
| Goods shipped | Sales → Finance | COGS ↑ / Inventory ↓ / AR ↑ / Revenue ↑ |
| Supplier paid | Finance | AP ↓ / Bank ↓ |
| Customer payment | Finance | AR ↓ / Bank ↑ |
| Purchase return | Purchase → Finance | AP ↓ / Inventory ↓ |
| Sales return (restock) | Sales → Finance | Revenue ↓ / AR ↓ / Inventory ↑ / COGS ↓ |
| Inventory adjustment | Inventory → Finance | Inventory ↑↓ / Inventory Adjustment Expense |
| Stock write-off | Inventory → Finance | Inventory Write-off Expense ↑ / Inventory ↓ |

### Accrual Accounting
- Transactions recorded in the period they are earned/incurred, not when cash moves
- Accrued expense entries created at period-end with auto-reversal in next period
- Prepaid expenses amortized over relevant periods
- All entries carry `fiscal_period_id` for period-locked reporting
